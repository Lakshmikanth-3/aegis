# Aegis

*Your agents pay in the open. What they spent, and why it was allowed, stays between you and the proof.*

## What this is

Stellar's x402 and MPP protocols now let AI agents autonomously discover, authorize, and settle payments for APIs, data, and compute — no human approving each transaction. That's live infrastructure, not a demo. But Stellar is a public ledger: every one of those payments — who an agent paid, how much, how often — is visible to anyone watching. For a company running a fleet of agents with department budgets and vendor relationships embedded in that spend pattern, the public spend graph is a competitive-intelligence leak, and today's spending controls (caps, allow-lists) are enforced by a contract anyone can inspect but no one can prove was configured correctly, or wasn't tampered with, without being handed the whole ledger.

Aegis closes both gaps with one circuit. Each agent draws from a shielded balance commitment instead of a visible running balance. Every spend is gated by a real Noir/UltraHonk zero-knowledge proof that checks the payment against policy — per-transaction cap, vendor allow-list, sufficient remaining balance — *before* the contract will let it settle; a non-compliant payment doesn't get rejected by an application-level check, it simply has no valid proof, because the circuit's constraints can't be satisfied. At any time, a treasury operator can generate a second, much smaller proof that discloses only an aggregate compliance fact — "this agent's spend over the last 24 hours stayed under $X, with zero payments to non-allow-listed vendors" — without disclosing a single underlying transaction.

## What's real vs. explicitly out of scope

**Real** (everything in the ZK + contract stack):

- `spend_proof` (`aegis-circuit/`): Poseidon commitments, a depth-3 Merkle vendor-allowlist membership check (8 vendors), per-transaction cap and balance constraints, nonce-based replay protection. 3/3 `nargo test` passing.
- `compliance_attestation` (`aegis-attestation-circuit/`): proves cumulative spend between two real commitment snapshots is bounded, without revealing individual amounts. 3/3 `nargo test` passing.
- `AegisTreasury` (`aegis-contract/`): deployed live on Stellar Testnet, verifying real UltraHonk proofs via the vendored [`rs-soroban-ultrahonk`](https://github.com/yugocabrio/rs-soroban-ultrahonk) verifier (MIT-licensed, vendored under `rs-soroban-ultrahonk/`, not written from scratch here) and Protocol 26 (CAP-80) BN254 host functions. Includes `set_period_start_commitment`, which lets the orchestrator attest against a genuine historical starting point (e.g. "last 24 hours") instead of only "since the last button click." 17/17 `cargo test` passing.
- Every proof is generated by real `nargo` (1.0.0-beta.9) + `bb` (v0.87.0, `--scheme ultra_honk --oracle_hash keccak`) — real 14,592-byte UltraHonk proofs, real 1,760-byte verification keys, not placeholder byte strings.
- Every payment rejection shown anywhere in the dashboard is a real circuit-level rejection: `nargo execute` fails to find a satisfying witness. This includes vendor-not-allowlisted rejections — the orchestrator constructs a syntactically valid but wrong Merkle proof attempt and submits it to the real circuit, rather than short-circuiting the check in JavaScript, so the real constraint solver is what rejects it every time.
- Every transaction hash shown in the dashboard is a real Stellar Testnet transaction, linked to stellar.expert.
- The orchestrator's off-chain Poseidon commitment math (`poseidon-lite`, used for speed) is cross-verified bit-identical against the real circuit's `poseidon::poseidon::bn254::hash_2/hash_3` output, captured live via `nargo test --show-output` (`orchestrator/src/selftest.ts`, 14/14 checks passing).
- The Vendor allow-list screen (`/vendors`): adding or removing a vendor rebuilds the depth-3 Merkle tree and issues a real `update_policy` transaction; the root shown always matches what's actually stored on-chain.
- The Live Sealed Feed never renders a plaintext amount, pass or fail. A settled payment's vendor also stays sealed; a rejected payment's vendor is shown, since a rejected payment never actually spent anything and naming it is useful audit transparency rather than a confidentiality leak.

**Explicitly out of scope** (a stated cryptographic boundary, not a shortcut):

- **Live x402/MPP facilitator integration.** The final settlement hop to a vendor's real address is, by construction, the one part of this design that must be publicly visible — like a Tornado-Cash-style privacy pool, deposits and internal transfers are private, but a withdrawal to an external address reveals an amount. Aegis hides *which* agent's shielded budget funded a payment and the link between an agent's successive payments, not the literal existence of a real payment rail at the very last hop. That final hop is logged, not executed against a live facilitator, in this build.
- **CAP-79 muxed sub-account agent identity.** Agent identity is a plain `u64` for this build, not a muxed `M...` sub-address under the treasury's funded `G...` account.

## Architecture

```
                          ┌──────────────────────┐
   AI Agent  ── wants to ─▶  Orchestrator          │
   (agent_id, vendor,      │  - real Poseidon math │
    amount)                │  - real nargo/bb       │
                           │    proof generation    │
                           └──────────┬────────────┘
                                      │ real nargo execute + bb prove
                                      │ (rejects here if no witness solves)
                                      ▼
                           ┌───────────────────────┐
                           │  AegisTreasury          │
                           │  (Soroban, Testnet)     │
                           │  - UltraHonk verifier   │
                           │    (CAP-80 BN254 fns)   │
                           │  - commitment + nonce   │
                           │    state per agent      │
                           │  - vendor allow-list    │
                           │    Merkle root          │
                           └──────────┬────────────┘
                                      │ AuthorizedSpendEvent
                                      ▼
                           ┌───────────────────────┐
                           │  x402 / MPP Facilitator │
                           │  (logged, not executed  │
                           │   in this build)        │
                           └───────────────────────┘

   Dashboard (Vite/React) ── talks to the orchestrator's REST + SSE API:
     /            Landing page (live settled/blocked stat row)
     /console     Treasury Console + per-agent detail drawer
     /feed        Live Sealed Feed (sealed amounts, real-time SSE)
     /vendors     Vendor allow-list management (real Merkle root updates)
     /attestation Compliance Attestation (24h / 7d / session, QR + independent verify)
```

Every arrow above is a real network call in this build except the bottom one (x402/MPP facilitator), which is the one explicitly out-of-scope boundary described above.

### Why this uses Stellar's newest ZK primitives

Protocol 25 ("X-Ray") added native Poseidon/Poseidon2 hashing, and Protocol 26 ("Yardstick", CAP-80) — which reached Testnet only weeks before this build — added the BN254 host functions that make UltraHonk proof verification cheap enough to gate a real payment on-chain rather than just being a research curiosity. Aegis exists because that primitive just landed: it's one of the first applications wiring UltraHonk verification into agent payments specifically, rather than generic private transfers.

| Suite | Command | Result |
|---|---|---|
| `spend_proof` circuit | `cd aegis-circuit && nargo test` | 3/3 passing |
| `compliance_attestation` circuit | `cd aegis-attestation-circuit && nargo test` | 3/3 passing |
| `AegisTreasury` contract | `cd aegis-contract && cargo test` | 17/17 passing (15 unit + 2 integration tests against real generated proofs) |
| Orchestrator Poseidon/Merkle self-test | `cd orchestrator && npm run selftest` | 14/14 checks passing |
| Dashboard Playwright e2e smoke test | `cd dashboard && npm run test:e2e` | 12/12 checks passing |

The e2e suite (`dashboard/e2e/smoke.mjs`) drives headless Chromium against the real running dashboard + orchestrator: real routing (`/`, `/console`, `/feed`, `/vendors`, `/attestation`), the real agent roster and drawer, a real 32-byte Merkle root on the vendor screen, and an explicit assertion that no payment amount ever appears in the DOM. It deliberately doesn't trigger new proof generation (that's covered by `demo-run.ts` and the contract's real-proof integration tests) to stay fast enough to run on every change.

## Running locally

Requires **WSL** (Ubuntu) — none of the Noir/Barretenberg/Stellar toolchain ships native Windows binaries. The orchestrator shells out to WSL automatically via `wsl.exe`, so you don't need to leave a Windows terminal day-to-day, but the toolchain has to exist inside WSL first:

```bash
# Noir, pinned to the version rs-soroban-ultrahonk's build script targets
curl -L https://raw.githubusercontent.com/noir-lang/noirup/main/install | bash
~/.nargo/bin/noirup -v 1.0.0-beta.9

# Barretenberg, pinned to v0.87.0 (not the default `bbup` install, which is newer/incompatible)
mkdir -p ~/.bb087/bin
curl -L https://github.com/AztecProtocol/aztec-packages/releases/download/v0.87.0/barretenberg-amd64-linux.tar.gz -o /tmp/bb.tar.gz
tar -xzf /tmp/bb.tar.gz -C ~/.bb087/bin

# Stellar CLI
mkdir -p ~/.local/bin
curl -L https://github.com/stellar/stellar-cli/releases/download/v27.0.0/stellar-cli-27.0.0-x86_64-unknown-linux-gnu.tar.gz -o /tmp/stellar.tar.gz
tar -xzf /tmp/stellar.tar.gz -C ~/.local/bin

# jq (bb's bytes_and_fields output post-processing needs it)
curl -sL https://github.com/jqlang/jq/releases/latest/download/jq-linux-amd64 -o ~/.local/bin/jq
chmod +x ~/.local/bin/jq ~/.local/bin/stellar

rustup target add wasm32v1-none   # for building the contract itself
```

Then:

```bash
# Circuits + contract
cd aegis-circuit && nargo test
cd ../aegis-attestation-circuit && nargo test
cd ../aegis-contract && cargo test

# Orchestrator (builds + deploys a fresh contract to Testnet, ~1-3 min)
cd orchestrator && npm install && npm run start   # http://localhost:4000
npm run seed                                       # registers the 5-agent / 8-vendor roster
npm run demo                                       # plays the 12-payment demo scenario
npm run selftest                                   # Poseidon/Merkle cross-check

# Dashboard
cd dashboard && npm install && npm run dev         # http://localhost:5173
npm run test:e2e                                   # real Playwright smoke test
```

`npm run start` (not `npm run dev`) is deliberate for the orchestrator during real use: dev mode watches files and re-triggers a full contract redeploy on every save, which is expensive and unnecessary once things are running.

## What we'd build next

- Live x402/MPP facilitator integration for the final settlement hop, instead of logging it.
- CAP-79 muxed sub-accounts for per-agent Stellar addresses.
- Per-agent (not treasury-wide) per-transaction caps — would need a small circuit extension to carry a per-agent cap as a public input bound to `agent_id`.
- Batched/parallel proof generation across agents — the orchestrator currently serializes `nargo`/`bb` calls per circuit to avoid clobbering shared `target/` build artifacts, which is the main bottleneck in a multi-payment demo run.
- A fuller Playwright e2e suite covering the real proof-generation and attestation-generation paths end to end, not just navigation and rendered-data assertions.
- Mobile-responsive dashboard layout.
